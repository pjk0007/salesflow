이거 없으면 "마케팅 분석 도구"라고 부르기 민망함.

### 1-1. 광고 클릭 ID 자동 추적 + 캠페인 ROI

**무엇인가**
URL에서 광고 식별자(`gclid`, `fbclid`, `n_media`, `kclkid` 등)를 자동 캡처해 방문자에 박아두고, "이 광고로 들어와서 가입한 사람 N명" 자동 집계.

**왜 필요한가**
- 우리는 지금 `utm_source/medium/campaign`만 봄 → 광고 플랫폼이 채워주는 진짜 광고 ID는 못 봄
- 광고 ROI 측정의 핵심. 없으면 "이 광고 효과 있나"에 답할 수 없음
- 디하·백오피스랩 모두 네이버/구글/메타에 광고 돌리고 있음

**장점**
- 작업량 작음 (tracker.js 50줄 + visitor 컬럼 몇 개 추가)
- 즉시 효과 (광고비 결정에 도움)
- 멀티 테넌시 자연 격리 (org별 광고)

**단점·주의**
- 광고 플랫폼이 파라미터 이름 바꾸면 따라가야 함 (예: Google이 `gclid` 외에 `gbraid`/`wbraid` 추가)
- iOS 추적 차단·쿠키 제한으로 일부 누락
- 광고 플랫폼 비용 데이터는 따로 가져와야 진정한 "ROI" (이건 또 별도)

**비용·인프라**
- 추가 비용 0
- 저장소 부담 미미 (visitor 컬럼만)

**범용성·멀티 테넌시 적합도**: ⭐⭐⭐⭐⭐
모든 마케팅 SaaS의 표준. 도메인 무관.

**참고**: 애디온이 이미 구현 ([tracker.js:60-160](../../adion/public/tracker.js))

### 1-2. 광고 플랫폼 데이터 자동 동기화 (큰 항목)

**무엇인가**
각 org가 자기 광고 계정(Google/Meta/네이버/카카오)을 우리한테 OAuth로 연결 → 우리는 그 토큰으로 광고 API 호출해서 **광고비·노출·클릭·전환 데이터**를 자동으로 끌어와 통합 분석.

**1-1과 차이**: 1-1은 visitor URL에서 광고 식별자 추출 (클라이언트, 가벼움), 1-2는 광고 플랫폼 API에서 실제 광고비·실적 가져옴 (서버, 무거움).

**우리 멀티 테넌시 구조에서의 그림**
```
샌드비
  ├── org 1: 디하
  │    ├── 구글 광고 계정 A 연동
  │    ├── 메타 광고 계정 B 연동
  │    └── 네이버 검색광고 계정 C 연동
  │
  ├── org 2: 백오피스랩
  │    ├── 구글 광고 계정 D 연동
  │    └── 메타 광고 계정 E 연동
  │
  └── org 3: 픽셀앤로직
       └── (아직 연동 안 함)
```

**왜 필요한가**
- 클릭ID만 추적하면 "광고로 들어와서 가입한 사람"은 알지만 **광고비 모름** → ROAS·CAC 계산 불가
- 광고비 + 전환 = "광고비 100만 / 가입 50 = 2만/가입 (CAC)" 측정 가능
- 운영자가 광고 매니저 안 들어가도 우리 대시보드에서 광고 성과 한눈에

**애디온의 "수동 동기화"가 왜 불편한가**
- 운영자가 매번 버튼 눌러야 → 까먹으면 데이터 stale
- 자동화의 가치가 사라짐
- 다만 애디온이 그렇게 만든 이유는 **운영 부담을 작게 시작하려고**임 — 이 항목이 진짜 무거운 이유는 아래 참조

---

#### A) 진짜 어려운 부분 — "분석은 쉬움, 데이터 끌어오기가 큰 산"

##### 1. OAuth가 4종류 다 다름
```
구글 광고  → Google OAuth 2.0 + Google Ads Developer Token (별도 신청·승인 필요)
메타 광고  → Facebook OAuth + Business Manager 권한
네이버    → API 키 (네이버 영업담당이 발급, 자동화 안 됨)
카카오    → Kakao OAuth + Business 계정 연결
```

각각 연동 화면도 따로따로. **연동 UI 4세트 + 토큰 보관 4세트 + 갱신 로직 4세트**.

##### 2. 권한 — 운영자가 광고 매니저 권한 있어야 OAuth 가능
운영 현실에서 가장 큰 장애물:

```
디하 운영팀 A가 샌드비에서 "구글 광고 연동" 클릭
   ↓
Google OAuth → A의 구글 계정 로그인
   ↓
"이 계정이 디하의 광고 계정 권한 있나?" 체크
   ↓
없으면 → 광고 매니저(보통 마케팅 팀장·CEO)에게 요청
   ↓
권한 받기 → 다시 OAuth → 비로소 연동 성공
```

→ 운영자가 광고 매니저 권한 있어야 하는데, 보통 마케팅 담당자만 있어서 **연동 시점에 권한 문의가 가장 많은 CS 문의**가 됨.

##### 3. 데이터 모델 — 광고 플랫폼마다 구조 다름

**구글 광고**: Customer → Campaign → AdGroup → Ad → Keyword
**메타 광고**: Ad Account → Campaign → AdSet → Ad (AdGroup 대신 AdSet — 의미 다름)
**네이버**: CustomerID → Campaign → AdGroup → Keyword + Ad
**카카오**: Account → Campaign → AdGroup → Ad

→ 통일된 추상화 만들어야 한 화면에서 비교 가능. 정규화 자체가 큰 설계 결정.

##### 4. 데이터 양 폭증
```
디하 광고 계정 1개:
  - 캠페인 50개
  - 광고 그룹 200개
  - 광고 소재 1000개
  - 키워드 5000개
  
하루치 metric (cost/impressions/clicks/conversions):
  → 약 6,000행 × 30일 × 12개월 = 216만 행/년
  
50개 org × 평균 2개 플랫폼:
  → 1년에 2억+ 행
```

인덱스·파티셔닝·집계 전략 필수.

##### 5. 토큰 만료 처리
```
OAuth 연동 → access_token (1시간) + refresh_token (90일)
   ↓
1시간마다 access_token 자동 갱신
   ↓
90일 후 refresh_token 만료 → 운영자가 다시 OAuth 해야 함
   ↓
"OAuth 재인증 필요" 알림을 운영자에게 알림톡·이메일로
```

운영 모니터링·재인증 알림 시스템 필수.

##### 6. Rate Limit
구글 광고 API는 1일 호출 한도 있음. 50개 org가 다 동시 동기화하면 한도 초과 가능.

→ **org별 시차 분산 (10분 간격 schedule) + 변경분만 가져오기** 설계 필수.

##### 7. 보안 — OAuth 토큰 = 광고 계정의 열쇠
refresh_token이 누출되면 다른 사람이 그 org의 광고 데이터 조회 가능.

→ DB에 저장 시 **AES-256 암호화 + 마스터 키 분리** 필수.

#### B) 동기화 방식 비교 (한 번 정리)

| 방식 | 동작 | 장점 | 단점 |
|---|---|---|---|
| **Pull-수동** (애디온) | 운영자가 버튼 누름 | 단순, 비용 통제 | 까먹음, 실시간 X |
| **Pull-자동** (Cron, 추천) | 15~60분마다 자동 호출 | 항상 최신, 운영자 무관여 | API 비용·rate limit 관리 필요 |
| **Pull-OnView** (똑똑) | 화면 진입 시 stale면 자동 새로고침 | 부담 적음, UX 좋음 | 구현 복잡 |
| **Push (Webhook)** | 광고 플랫폼이 push | 실시간, 비용 적음 | **광고 플랫폼이 지원해야 함 — Google/네이버는 미지원** |

**플랫폼별 현실**: Google/네이버/카카오는 Pull만, Meta만 부분 Push.
→ 사실상 **Pull-자동 (Cron) + 수동 버튼** 이 표준.

**권장 아키텍처 (Hybrid Pull)**:
```
1. Cron 자동 동기화 (org별 시차 분산)
   - 캠페인 메타: 1시간마다
   - 비용·노출·클릭: 30분마다
   - 전환: 15분마다

2. 운영자 "지금 새로고침" 수동 버튼
   - rate limit (1분 1번)

3. UI에 "마지막 동기화: N분 전" 표시
4. 토큰 만료 임박 시 운영자에게 알림 (알림톡 API 활용)
```

### 1-3. 세션 리코딩 (Session Replay)

**무엇인가**
사용자가 페이지에서 실제로 한 동작을 영상처럼 재현. 클릭·스크롤·입력·DOM 변경 다 캡처. 표준 라이브러리는 `rrweb` (MIT 오픈소스).

**왜 필요한가**
- 마케팅·UX팀이 가장 원하는 기능 1순위
- 데이터에 안 잡히는 "왜?"의 답
  - "구독 직전에 이탈한 visitor가 정확히 뭘 봤나"
  - "버튼이 안 눌리는 이유가 디자인 문제인가 버그인가"
- Hotjar/FullStory/PostHog의 대표 기능

**장점**
- rrweb 자체는 **MIT 무료**, 외부 SDK 의존 없음
- 마케팅·UX·CS·QA·개발 모두에 도움
- 한 번 인프라 만들면 다양한 분석 파생 가능 (히트맵·rage click 등)

**단점·주의 (중요)**
- **인프라 부담 큼**
  - rrweb 페이로드는 **5분 세션 = 100~500KB** (압축 후)
  - 10만 세션/월이면 **DB 또는 S3 저장소 5~10GB/월**
  - 검색은 안 됨 → 메타데이터(visitor·시간·url)만 따로 인덱싱
- **민감 정보 마스킹 필수**
  - 비밀번호·결제 정보·이메일 등 — rrweb의 `maskAllInputs`/`blockClass` 옵션
- **재생 UI 구현 비용**
  - 그냥 캡처만으론 부족. 재생기 UI까지 따로 만들어야 함
- **녹화 가능성·법적 이슈**
  - 한국 개인정보보호법 측면에서 동의 필요할 수 있음 (GDPR 권역이면 더)
- **성능 영향**
  - 클라이언트 CPU·메모리 약간 부담 (rrweb 잘 만들어져 있어서 미미하긴 함)

**비용·인프라 (현실적)**
- rrweb 라이브러리: **무료** (MIT)
- S3 저장소: **약 $5~10/월 per 10만 세션 (자체 호스팅 기준)**
- 데이터베이스 추가 부담: 세션 메타데이터만 우리 DB, 본문은 S3
- 운영 비용 (Hotjar 같은 SaaS 쓰면): **$80~$300/월부터**
- → **우리는 자체 구현 추천** (rrweb 오픈소스라 가능)

**범용성·멀티 테넌시 적합도**: ⭐⭐⭐⭐
모든 org에 가치. 다만 저장소 비용을 어떻게 분배할지(사용량 기반 요금?) 고민 필요.

**참고**: [rrweb GitHub](https://github.com/rrweb-io/rrweb), [PostHog 구현](https://posthog.com/docs/session-replay)

### 1-4. Multi-touch Attribution

**무엇인가**
"한 명이 가입까지 오는 데 거친 모든 채널"에 전환 credit을 나눠 배정.

```
방문자 흐름:
  1차 방문: 구글 광고 (gclid)
  2차 방문: 네이버 검색
  3차 방문: 이메일 클릭
  4차 방문: 직접 입력 → 가입

[First Touch]    구글 광고 100%
[Last Touch]     직접 입력 100%
[Linear]         구글 광고 25% + 네이버 25% + 이메일 25% + 직접 25%
[U-shape (40-20-40)] 구글 광고 40% + 가운데 20% + 직접 40%
[Time-decay]     최근일수록 가중치 ↑
```

**왜 필요한가**
- B2B는 사이클이 길어서 (디하 평균 며칠? 매치 → 결제) 한 번에 안 옴
- "라스트 클릭만 보면 광고에 돈 안 쓰게 됨" → 진실 왜곡
- 광고 예산 분배 결정에 핵심

**장점**
- 실제 데이터로 정확한 광고 ROI 계산
- 4~5개 모델 제공해서 운영자가 선택
- 우리는 이미 tracker_sessions에 referrer/utm 시간순 다 저장 중 → **데이터 인프라는 거의 다 있음**, 알고리즘만 얹으면 됨

**단점·주의**
- 알고리즘 복잡도 중간 (특히 W-shape, 시간 감쇠)
- 운영자 학습 곡선 (어느 모델이 우리 도메인에 맞는가)
- 멀티 디바이스 + 멀티 브라우저 추적 어려움 (visitor merge 필요)

**비용·인프라**
- 추가 비용 0
- 저장소 부담 적음 (계산은 분석 시점)
- 계산 비용 약간 ↑ (한 visitor의 모든 세션 join)

**범용성·멀티 테넌시 적합도**: ⭐⭐⭐⭐⭐
B2B 마케팅의 표준 분석. 모든 org에 가치.

**참고**: [Multi-touch Attribution 가이드](https://improvado.io/blog/multi-touch-attribution-solutions)

## 2. Layer 2 — 강력 (있으면 다른 도구 대체)

### 2-1. Cohort Retention

**무엇인가**
"5월에 가입한 사람들이 1주 후·1개월 후·3개월 후 잔존한 비율" 매트릭스.

```
가입주차   1주    2주    3주    4주
2026-05    100%   80%   65%   50%
2026-06    100%   85%   70%   ?
```

**왜 필요한가**
- B2B SaaS의 **가장 중요한 long-term 지표**. PostHog/Amplitude 다 강조
- 마케팅 채널별 코호트 비교 → "어느 채널이 진짜로 충성 고객 데려오나" 답

**장점**
- 우리는 record_events 다 있어서 데이터 인프라 완성됨
- 알고리즘 표준 (SQL DATE_TRUNC + window 함수)

**단점**
- 코호트 정의가 복잡할 수 있음 (운영자 학습 필요)
- 시각화 UI 만들기 살짝 까다로움 (heatmap 형태)

**비용·인프라**: 추가 비용 0, 계산 약간 ↑
**적합도**: ⭐⭐⭐⭐⭐ — SaaS 표준

---

### 2-2. 퍼널 v2 (Drop-off + 채널 분해 + Revenue)

**무엇인가**
우리 퍼널 v1을 다음으로 강화:

1. **"최종 전환 단계" 마크**: 운영자가 퍼널 단계 중 하나를 ⭐ 표시 → 그게 그 사이트의 단일 KPI
2. **채널 × 단계 매트릭스**: "네이버 광고로 들어온 사람들의 단계별 전환율"
3. **Drop-off 분석**: 단계간 평균 소요 시간 + 멈춘 visitor 명단
4. **Revenue (선택)**: 단계에 매출 값 매핑 → 채널별 매출·ROAS 계산
5. **명단 retargeting**: 멈춘 사람들 명단을 알림톡 발송 API로 바로 보냄

**왜 필요한가**
- 현재 퍼널은 "몇 명이 거쳤나"만. 진단·액션·전환 KPI 정의 모두 빠짐
- "어디서 막히나" + "어느 채널이 진짜 매출 만드나"가 마케팅의 두 핵심 질문
- 별도 Conversion Goals 도구를 두지 않고 퍼널 안에서 다 해결 (학습 비용 ↓)

**장점**
- 우리 퍼널 v1 위에 자연스럽게 얹힘
- 한 도구에서 단계별 흐름 + 단일 KPI + retargeting 다 됨
- 알림톡 API와 시너지 (멈춘 사람 발송)
- record + tracker 통합 데이터 활용 가능 (우리만의 강점)

**단점**
- 퍼널 정의 UI가 복잡해짐 (전환 마크, revenue 입력 등)
- visitor 명단 SQL 무거움 (페이지네이션 필수)
- "마지막 단계 = 전환"이 항상 자연스럽지 않을 수 있음 (B2B에서 "구독" 다음에 "이탈"도 있음)

**비용·인프라**: 0
**적합도**: ⭐⭐⭐⭐⭐ — 우리만의 강점 (record + tracker 통합 데이터)

**참고**: 애디온 `trackerConversionGoals` 테이블이 분리형이었는데, 우리는 통합형으로 가는 선택.

### 2-3. Click & Scroll Heatmap

**무엇인가**
페이지 위에 사용자 클릭/스크롤 분포를 색깔 점/그라데이션으로 시각화.

```
[페이지 미리보기 위에]
🔴🔴 hero CTA — 가장 많이 클릭
🟡    pricing — 중간
⚪    faq    — 거의 안 봄
```

**왜 필요한가**
- 우리 인게이지먼트는 섹션 단위인데, "그 섹션의 어디를 정확히 클릭했나"는 없음
- UX 개선 결정의 직관적 도구

**장점**
- 매우 시각적 (마케터·디자이너 친화)
- 클라이언트는 단순 (좌표만 기록)

**단점·주의**
- 페이지 스크린샷 생성 필요 (cron으로 정적 캡처 / 또는 운영자가 업로드)
- 반응형이라 viewport 크기별로 따로 봐야 함
- 데이터 양 많음 (클릭마다 좌표 + viewport)

**비용·인프라**
- 추가 비용 약간 (스크린샷 생성 cron + 저장)
- 저장 부담 중간 (클릭 좌표 누적)

**적합도**: ⭐⭐⭐ — 멀티 테넌시는 OK지만 인프라 부담

## 3. Layer 3 — 보강 (운영 편의)

### 3-1. Alert / Notification 시스템

**무엇인가**
"전환율 30% 하락", "신규 가입 1시간 0건" 같은 임계값 룰 기반 알림. 슬랙·이메일·알림톡으로 발송.

**장점**
- 우리 알림톡 API와 즉시 연결 (시너지)
- 일별 보고서 자동 발송 가능

**단점**
- 룰 엔진 만들어야 함 (조건 표현식)
- false positive 관리 (너무 많이 울리면 무시됨)

**비용·인프라**: 추가 비용 0
**적합도**: ⭐⭐⭐⭐ — org별 룰 자유

### 3-2. 이메일·알림톡 캠페인 통합 분석

**무엇인가**
보낸 메시지 → 클릭 → 사이트 방문 → 전환 → 매출까지 단일 화면.

**왜 필요한가**
- 우리는 email_send_logs / alimtalk_send_logs / tracker_events 다 있지만 **한 화면에서 join 안 됨**
- 캠페인 ROI 측정의 결정적 누락

**장점**
- 우리 시스템만의 강점 (3개 도메인 다 들고 있음)
- 마케터의 일상 질문에 직접 답

**단점**
- SQL 복잡 (3개 테이블 join + 시간 윈도우)
- UI 복잡도 ↑

**비용·인프라**: 0
**적합도**: ⭐⭐⭐⭐⭐ — 우리만의 강점

### 3-3. UTM Builder + Short Link

**무엇인가**
- 캠페인 URL 생성기 (utm 자동 채움, 미리보기, 복사)
- 짧은 URL 발급 (`sendb.kr/r/abc123`) + 클릭 카운트

**장점**
- 운영 편의 큼
- 마케터의 일상 작업 도구

**단점**
- 핵심 기능 아님 (사실 외부 도구도 많음 — Bitly 등)
- 도메인 SEO·신뢰도 영향 살짝

**비용·인프라**: 0
**적합도**: ⭐⭐⭐ — 있으면 좋음

### 3-4. 사용자 정의 대시보드 (위젯 빌더)

**무엇인가**
운영자가 보고 싶은 차트를 직접 드래그·드롭으로 조합.

**장점**
- 운영자 만족도 매우 ↑ (Mixpanel/Amplitude 강점)
- org별 핵심 KPI 다 다름

**단점**
- 구현 매우 복잡 (드래그&드롭 + 차트 라이브러리 + 스토리지)
- 운영자 학습 곡선 큼

**비용·인프라**: 0
**적합도**: ⭐⭐⭐ — 가치 큰데 구현 부담 큼 (장기)

## 4. Layer 4 — 도메인 특화·실험적

### 4-1. Path Analysis (자동 여정 발견)
- "방문자들이 가장 많이 거치는 5개 페이지 경로"
- Markov chain 또는 단순 sequence mining
- 우리 record_events + tracker_events 통합 가능

### 4-2. AI 기반 인사이트
- "이번 주 가입 떨어진 이유: 네이버 광고 클릭률 -40%"
- LLM에 데이터 요약 + 자연어 설명
- PostHog 2026 추세

### 4-3. Anomaly Detection
- 자동 임계값 감지 (이전 30일 대비 ±N σ)
- Alert 시스템과 연동

## 5. 추천 — 도입 순서

**가성비 안 따지고 "있으면 우리한테 진짜 도움" 기준**:

```
Phase 1 (즉시 임팩트, 인프라 가벼움)
├── 1-1. 광고 클릭ID 자동 추적
├── 2-2. 퍼널 v2 (Drop-off + 채널분해 + Revenue)  ← 전환 KPI 흡수
└── 1-4. Multi-touch Attribution

Phase 2 (강력한 분석, 운영 부담 중간)
├── 1-2. 광고 플랫폼 데이터 동기화      ← OAuth + Cron 인프라
├── 2-1. Cohort Retention
└── 3-2. 이메일·알림톡 캠페인 통합

Phase 3 (인프라 부담 큼)
├── 1-3. 세션 리코딩 (rrweb)
└── 2-3. Heatmap

Phase 4 (장기 — 큰 작업)
├── 3-1. Alert 시스템
└── 3-4. 사용자 정의 대시보드

Phase 5 (실험)
└── 4-* (Path Analysis, AI 인사이트 등)
```

**Phase 1**은 "이거 없으면 마케팅 도구 아님" 수준의 필수.
- 퍼널 v2가 단일 KPI(전환 목표) 역할도 흡수해서 별도 사이클 없이 한 도구로 통합
**Phase 2**는 "이거 있어야 진짜 인사이트 나옴".
**Phase 3**부터는 인프라·운영 비용 고려 필요.
